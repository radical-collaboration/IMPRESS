● Each record in the ensemble store has input_ref pointing to the output_ref of
  the preceding step in the same target's trajectory. For protein_a, pass 1:

  ┌──────────┬───────────────────────────────┬───────────────────────────────┐
  │   type   │           input_ref           │          output_ref           │
  ├──────────┼───────────────────────────────┼───────────────────────────────┤
  │ backbone │ START                         │ backbone_protein_a_pass1_2659 │
  ├──────────┼───────────────────────────────┼───────────────────────────────┤
  │ sequence │ backbone_protein_a_pass1_2659 │ CCDKGAEWSNTRVKKGHPQR          │
  ├──────────┼───────────────────────────────┼───────────────────────────────┤
  │ decoy    │ CCDKGAEWSNTRVKKGHPQR          │ /…/best_models/protein_a.pdb  │
  └──────────┴───────────────────────────────┴───────────────────────────────┘

  What each link means:

  - START → backbone_ref: The backbone was predicted from nothing (or from the
  initial PDB input). "START" is the sentinel value analyze_backbone hard-codes
  as input_ref for every backbone record because there's no prior step in the
  trajectory to reference.
  - backbone_ref → sequence_prefix: analyze_sequence reads
  state.backbone_refs[target] as input_ref. This records that the sequence
  design was conditioned on that specific backbone structure. The output_ref is
  the first 20 characters of the top-ranked sequence — a compact but non-unique
  identifier (it's a prefix, not a hash).
  - sequence_prefix → decoy_pdb_path: analyze_fold reads
  state.selected_sequences[target][:20] as input_ref. This records that the fold
   prediction was run on that specific sequence. The output_ref is the absolute
  path to the PDB file written by write_mock_alphafold_for_target.

  Why "the chain is intact" matters:

  The chain being intact means the provenance graph is reconstructable from the
  JSONL alone — you can trace any decoy back through the sequence that produced
  it and the backbone it was designed on. If analyze_backbone had used a stale
  state.backbone_refs (e.g., from a prior iteration if trajectory state was
  leaking between passes), or if analyze_sequence had used the wrong target's
  sequence, the input_ref links would be broken or mismatched. The fact that
  sequence.input_ref == backbone.output_ref for the same target in the same pass
   means LangGraph state propagation is correct and no cross-target or
  cross-pass contamination occurred.

  What's still imprecise:

  The sequence output_ref is a 20-character prefix, not a unique identifier. Two
   different sequences could share the same prefix. For the mock, this is fine —
   the sequences are long and deterministic — but in a real pipeline the
  output_ref for a sequence step should be a hash or a file path to the .fa
  file, not a truncated string.
